How GitOps works in platform

platform delivers software by making the git repository the single source of truth: every desired-state fact — including the container image ref — lives committed in CUE, and Flux reconciles it onto the cluster. Two phases: bootstrap the appliance once, then run the steady-state deploy loop forever.

Flux only ArgoCD — deprecated Keel — deprecated

The one rule: render is a pure function of the committed git tree (apps/*.cue + platform.toml). Nothing that decides what runs — least of all the image ref — is injected at render time. To change what's deployed, you change git. Flux is the sole GitOps engine; ArgoCD and Keel are gone.

Phase 1 — Bootstrap the appliance

platform is an opinionated appliance: it ships the cluster setup, not a toolbox. Onboarding a fresh infra repo writes three things into git…

infra-repo/ platform.toml # [ops.vars] — env-style version pins (NGINX_GATEWAY_VERSION, …) cue.mod/ # pins prodigy9.co/defs apps/ # *.cue app definitions + *.platform baseline directives

…and installs the embedded baseline onto the cluster. These four are platform's opinion — fixed, and version-locked with the tool:

What lands on the cluster

  • cert-manager — issues TLS.
  • Flux — the in-cluster GitOps reconciler. From here on Flux is what applies desired state; nothing is kubectl apply-ed by hand.
  • NGINX Gateway Fabric — Gateway-API ingress.
  • Dagger engine pool — a privileged StatefulSet of BuildKit daemons, the in-cluster build substrate. platform finds the pods through the headless Service's DNS A-records and round-robins build sessions across them.

Cold-start has no unbreakable cycle: the engine ships as plain manifests platform applies directly, and the first platform image is built by a local auto-provisioned engine. Once the pool is up, platform builds on its own engines.

Phase 2 — The deploy loop

Once the appliance is up, every deploy is the same loop. The git commit is the gate; everything downstream is mechanical and reproducible from it.

  1. platform build — Dagger builds the app's container on the in-cluster engine pool and pushes it to GHCR under an immutable ref (digest / release tag).
  2. commit the ref — that immutable ref is written into the infra repo's apps/*.cue as a committed literal, and committed. This commit is the deploy.
  3. platform ops render — exports the apps CUE package and applies the .platform directives into a manifest tree. A pure function of the committed tree: the image is read from the literal in CUE, never injected.
  4. platform ops publish — packs the manifests as an OCI config artifact (Flux media types) and pushes it to GHCR under a moving per-env tag.
  5. Flux reconcile — an OCIRepository watches that tag, pulls the artifact, and applies it. The cluster converges to exactly what git holds.

The loop closes on git: the next change is another commit, and Flux continuously drives the cluster back to the committed state.

Why the image ref lives in git

The image is desired state. If it entered at render time from a flag or env, the git source couldn't tell you what's deployed and a deploy couldn't be reproduced from the tree alone — the very thing GitOps exists to remove.

So the ref is a concrete value in apps/*.cue: deploying is a git commit, render and publish are pure functions of the tree, and Flux reconciles exactly what's committed.

Operator settings: [ops.vars]

Versions and similar knobs the operator owns live in platform.toml under [ops.vars] — also committed, so they too are desired state. They reach the baseline hooks by a single convention:

# platform.toml — env-style keys, the preferred form [ops.vars] NGINX_GATEWAY_VERSION = "v2.6.0" # consumed lowercase-derived by both render routes: # CUE apps/*.cue → #version: string @tag(nginx_gateway_version) # DSL *.platform → download "…/\(nginx_gateway_version)/…"

Same git rule: var values come only from committed platform.toml, never an environment variable at render time. platform attaches no meaning to any var — it is a pure name→value passthrough.
