## The engine model (current design-of-record — supersedes the Slices 3–4 shape)

The Slices 3–4 design (`builder.Engine` interface + `engine.Pool` + ctx-storing session) was
**reworked this session**. Current shape:

- **No `builder.Engine` interface** — it was introduced in Slice 3 (`cec052b`) then removed
  (`84a10a5`). It was never the cycle-breaker (*"builder doesn't import engine"* is), and
  `*dagger.Client` comes from `dagger`, not `engine`, so builders take it concretely:
  `Interface.Build(ctx, *dagger.Client, unit)`. Graph still `engine → builder → project`.
- **`engine.Engine` is the fleet handle** (sql.DB analog): process-wide, concurrency-safe,
  connections dialed lazily and reused. Built with `New(cfg *fxconfig.Source)`.
- **Context-carried like `fx/data`** — `NewContext`/`FromContext`/`LookupFromContext`;
  `Build(ctx, attempt)` / `Publish(ctx, builds...)` resolve the engine *from* ctx; context
  flows from the caller, never stored in the handle.
- **Three single-purpose units:** `discovery` (stateless DNS, the only config reader),
  `clients` (one reused `*dagger.Client` per endpoint, **validate-on-borrow** via a cheap
  `Version()` ping → redial if dead; holds the only lock, never across I/O), `Engine` core
  (round-robin `atomic.Uint64` cursor, no lock of its own).
- **No environment detection.** `DAGGER_ENGINE` (DNS name) + `DAGGER_ENGINE_PORT` (default
  1234) drive discovery: set → resolve endpoints; unset → empty. The `KUBERNETES_SERVICE_HOST`
  sniff and hardcoded service/namespace consts are **gone**. "Empty discovered → use the local
  engine" is an explicit branch in `Engine.Client`; the local-dial mechanism (Connect with no
  runner host) lives in `clients.dialEngine` (empty host = local).
- **`BuildResult.client`** carries the building engine for Publish reuse + preview's tunnel
  (`BuildResult.Client()`); **`Engine.Client(ctx)`** is the raw-access escape hatch for
  `ls`/`preview`.

The `BuildAttempt ─has many→ BuildUnit` model (one invocation = one attempt; resolved
`BuildUnit.Builder` value, not a name string) is unchanged and still correct.

## Forward design: platform-as-CI (the payoff of the engine reshape)

The engine is now exactly the executor a server needs: `engine.New(cfg)` once at startup,
inject it per request via `engine.NewContext(req.Context(), eng)`, and a webhook handler is
pure delegation — receive push → clone → `project.Configure(path)` → `builder.AttemptFrom` →
`engine.Build(ctx, attempt)`. In-cluster it round-robins the engines named by `DAGGER_ENGINE`
(the deployment sets that env var when the server lands; until then unset → local). The
infra-repo case (render+publish) is where a container-run builder *could* earn its keep
(per-repo platform-version pinning) — deferred, not decided.

## Key facts (still current)

- **publish auth:** `platform publish` uses the LOCAL docker ghcr creds (osxkeychain) — no
  `REGISTRY_USERNAME`/`PASSWORD` needed locally.
- **arch crash:** a publish through the local Dagger engine on an arm Mac builds arm64; it
  exec-crashes (exit 255) on amd64 LKE nodes. `publish_arch` (default amd64) fixes it.
- **digest pin:** a rebuilt tag (arm→amd64) leaves nodes caching the old digest;
  `IfNotPresent` won't re-pull. Pin `tag@sha256` to defeat the stale cache.
- **DSL verb is `focus`, not `select`;** there is no `[field=val]` selector (filter lives in
  `focus`).
