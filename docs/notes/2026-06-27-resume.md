## Forward design: platform-as-CI (lay out in a thinking session next time)

chakrit's sketch for the CI/CD component that drives fleet delivery (the thing the
non-platform fleet migrates onto, per the migration plan's forward direction). Not
settled — think it through next session:

1. A **GitHub webhook receiver** (an HTTP server — the vanity server is the natural host).
2. On push, **git clone** the updated repo.
3. Run the **"engine"** (the same build pipeline) — but instead of via a CLI command,
   **trigger it inside the webhook's HTTP request context**.
4. That context has special setup: it points at the **in-cluster hosted Dagger engine**
   (`dagger-engine.platform`) rather than the local docker-socket one. (Ties to
   `builder/engine.go`'s in-cluster pool discovery — already exists.)
5. **Infra image → Flux update:** reuse the *exact same flow*, just with an infra repo
   instead of a code repo. Possibly model `platform-infra` as **just another kind of
   `builder/`** that calls `render` internally and packages/publishes the Flux OCI image —
   maybe via plain shell scripts, no new platform feature surface.

This is the natural payoff of the jobs-as-data + `engine/` split (queued #1): once the
engine is a clean stage you pass a `Job` to, triggering it from an HTTP handler vs a CLI
command is the same call with a different engine target.

## Key facts learned

- **publish auth**: `platform publish` uses the LOCAL docker ghcr credentials (osxkeychain)
  — no `REGISTRY_USERNAME`/`PASSWORD` needed locally. (chakrit's hunch; verified.)
- **arch crash**: a publish through the *local* Dagger engine on an arm Mac builds arm64;
  it exec-crashes (exit 255) on the amd64 LKE nodes. `publish_arch` (default amd64) fixes it.
- **digest pin**: when a tag is rebuilt (arm→amd64), nodes cache the old digest under that
  tag; `IfNotPresent` won't re-pull. Pin `tag@sha256` to defeat the stale cache.
