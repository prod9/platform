## The settled design (drives Slices 3–4)

Re-derived in conversation; encoded in `architecture.md`. Key reversals vs the old spec:

- **`BuildAttempt` ─has many→ `BuildUnit`.** One invocation (CLI `build`, future webhook
  push) = one attempt; one `BuildUnit` per selected module (the old `Job`).
- **`BuildUnit.Builder` holds the RESOLVED `Builder` value, not a name string.** The old
  spec said "holds a builder name (string)" and claimed the string broke the import cycle —
  **both wrong.** A string buys nothing (the engine would re-`FindBuilder` either way), and
  the cycle is broken differently (below). The unit is an in-memory arg-bag, not a record.
- **Persistence is a future db-package's concern**, not `builder`/`engine`'s. When a build
  is ever persisted, that db package owns a shim/record type mapping *from* `BuildUnit`;
  the model never contorts to be row-shaped.
- **The cycle-breaker is a consumer-side `Session` interface.** `builder` declares a tiny
  `Session interface { Client(); Context() }`; the strategies consume *that*; `engine`'s
  concrete `Pool` satisfies it implicitly. Final graph `engine → builder → project`,
  acyclic, because `builder` never imports `engine`.
- **`engine/` is the executor** (the `Build`/`Publish` run-loop + the Dagger `Pool`);
  `builder/` is the strategies/knowledge + the `BuildAttempt`/`BuildUnit` data + the
  `Session` interface. (Builders hold irreducible per-stack logic — Go-version fetch,
  corepack, asset/build-dir detection — so they are NOT reducible to declarative step data;
  a "list of steps" model was explicitly rejected.)
- **Two models, both data:** `Project`/`Module` (config) and `BuildAttempt`/`BuildUnit`
  (resolved) are both data (source vs IR); behavior lives in the strategies + engine.

## Forward design: platform-as-CI (the payoff of the engine split)

Once `engine/` is a clean executor you hand a `BuildAttempt` to, triggering it from an HTTP
handler vs a CLI is the same call with a different engine target. The webhook flow is pure
delegation: receive push → clone repo → `project.Configure(path)` → `builder.AttemptFrom` →
`engine.Build` on the in-cluster `dagger-engine.platform` pool (already auto-discovered by
`engine.discoverEngines` via `KUBERNETES_SERVICE_HOST`). No new structure needed for code
repos. The infra-repo case (render+publish) is where a container-run builder *could* earn
its keep (per-repo platform-version pinning) — deferred, not decided.

## Key facts (still current)

- **publish auth:** `platform publish` uses the LOCAL docker ghcr creds (osxkeychain) — no
  `REGISTRY_USERNAME`/`PASSWORD` needed locally.
- **arch crash:** a publish through the local Dagger engine on an arm Mac builds arm64; it
  exec-crashes (exit 255) on amd64 LKE nodes. `publish_arch` (default amd64) fixes it.
- **digest pin:** a rebuilt tag (arm→amd64) leaves nodes caching the old digest;
  `IfNotPresent` won't re-pull. Pin `tag@sha256` to defeat the stale cache.
- **DSL verb is `focus`, not `select`;** there is no `[field=val]` selector (filter lives in
  `focus`). The code is correct; docs were stale (now fixed except CLAUDE.md).
