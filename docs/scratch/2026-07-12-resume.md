5. **Flux webhook delivery — DESIGN CAPTURED (buildable now — not server-gated).** The
   ingress-direct model means instant delivery needs only the baseline Receiver/ingress +
   platform's webhook-config step; the platform server is not on the delivery path.
   Corrects a model error: **pull ≠ poll**. "Pull model" only means Flux runs in-cluster and pulls state; the
   *trigger* is webhook, instant on image upload — exactly what ArgoCD (prior model) always
   did. **"tag-watch" is a valid term and STAYS** — Keel realized it via *registry webhook*
   (instant), never polling; the only error was a prior session (and me) reading watch=poll.
   No doc purge. Design:
   - Baseline ships a **`github`-type Flux `Receiver`** (not `generic-hmac` — the
     GitHub-specific type does native `X-Hub-Signature` HMAC + `spec.events` filtering) on
     event `registry_package` (ghcr image upload), plus an **ingress** exposing it.
   - **platform configures the GitHub repo webhook** (GitHub API, needs `admin:repo_hook`)
     to POST at that ingress — new platform capability, likely in infra `init`/setup. Do
     NOT re-add a "verify `registry_package` fires" step: Keel consumed exactly that event
     in production — it's observed, not a question.
   - **Delivery path evolved:** first cut was a *platform-server broker* (server receives
     GitHub webhooks, fans out to Flux). Operator superseded it with **ingress-direct**:
     GitHub → Flux receiver directly, no broker in the path. Resilience win — server down ⇒
     local `platform publish` (build+publish) still delivers, because GitHub→Flux doesn't
     route through the server.
   - **Two distinct webhook flows / two hostnames** (don't merge them): `FLUX_HOSTNAME` =
     the Flux receiver ingress (GitHub `registry_package` → reconcile). `PLATFORM_HOSTNAME`
     = the platform server's *own* ingress — the server is the automated publish-driver
     (receives its build-triggering webhooks), upstream of publish, not in the Flux path.
     Both join baseline `DefaultVars`, land wherever item 3 puts the vars table.
   - **Intervals stay SHORT** — webhook and poll **coexist**. The `OCIRepository.interval`
     (source-fetch, currently `1m`) and `Kustomization.interval` (cluster drift-repair,
     `10m`) are the *dropped-webhook fallback*; a long interval would degrade the safety
     net. Webhook = instant common case, interval = bounded worst case. Mechanism: the
     webhook reconciles the OCIRepository (source); the Kustomization then cascades an apply
     because its source changed.
   - Files in play: `framework/baseline/apps-flux-sync.cue.tmpl` (add Receiver; keep
     intervals), `framework/baseline/apps-flux.platform`, `framework/baseline.go`
     (`DefaultVars`).
   - **Flag (verify, don't assume):** `docs/spec/platform-server.md` predates this
     refinement — reconcile it with the ingress-direct model (server is publish-driver, not
     the delivery path) when building this. The "tag-watch" term there STAYS (it's correct =
     instant/webhook); only check the mechanism prose doesn't imply polling or server-in-path.

6. **Anti-regression guard for item 5 — CAPTURED.** The regression to prevent is **webhook
